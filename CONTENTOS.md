## Execution mode

CONTENT:OS supports two execution modes, configured per workspace in `workspace.config.md`:

### Interactive mode (default)
- Confirms each major decision with an approval gate
- Shows full context before proceeding
- Pauses at every checkpoint for user input
- Best for: new workspaces, learning the system, high-stakes content

### Auto mode
- Auto-approves most decisions вҖ” just executes
- Skips approval gates for content drafts, brief approvals, review results, optimization suggestions, and repurpose versions
- Still shows results inline so you can review, but does not pause
- Only stops for **hard gates** (non-skippable):

  **Outbound (audience sees it вҖ” irreversible):**
  - `/content:publish` вҖ” pushing live content to any channel
  - Social media posting вҖ” publishing to any social platform
  - Cross-platform distribution вҖ” syndicating content externally
  - Public announcements вҖ” press releases, launch posts
  - Newsletter sends вҖ” email distribution to subscriber lists

  **Data integrity (corrupts your content pipeline):**
  - Brand voice file edits вҖ” changes to BRAND.md, TOV.md, STYLE.md (these define all content direction)
  - Content deletion вҖ” removing published or validated content
  - Editorial calendar overwrites вҖ” modifying scheduled content plans
  - Distribution list changes вҖ” modifying subscriber lists or segments

  **Infrastructure (breaks your publishing):**
  - CMS config changes вҖ” publishing platform settings, templates, permissions
  - Social account connections вҖ” adding, removing, or modifying connected accounts
  - API key or credential changes вҖ” rotating, updating, or exposing keys in .env
  - Webhook creation/deletion вҖ” webhooks push data to external systems
  - SEO config changes вҖ” robots.txt, sitemap, canonical URL, redirect rules

  **Financial / compliance:**
  - Budget overages вҖ” tool spend exceeds workspace budget
  - Compliance/legal violations вҖ” copyright, licensing, legal claims
  - Tool credit checks marked `confirm-before-every-use`

**How it works in commands:**
- Commands that show `>> Approve / Edit / Reject` gates: in auto mode, auto-approve and continue. Log the auto-approval in `logs/decisions.md`.
- Commands that ask clarifying questions: in auto mode, use sensible defaults from `defaults.md` and proceed. Log what was assumed.
- Multi-step workflows (brief вҶ’ write вҶ’ review вҶ’ publish): in auto mode, chain automatically. Stop only at hard gates.

### Audit log

Every action in auto mode вҖ” not just gate decisions вҖ” gets logged to `logs/auto-audit.md`. This is the "black box" that lets you trace what happened if something goes wrong.

**Log every auto-mode action with:**
```
## [ISO timestamp]
- **Action:** what was done (e.g. "Generated 3 LinkedIn post variants")
- **Tool:** which tool/API was called
- **Input:** key parameters (endpoint, record count, query)
- **Output:** result summary (records returned, status, errors)
- **Cost:** credits/units consumed
- **Files changed:** which files were created or modified
- **Auto-approved gate?** yes/no вҖ” if yes, what gate was skipped
```

Keep this log append-only. Never truncate or overwrite. Rotate monthly to `logs/auto-audit-YYYY-MM.md`.

### Circuit breakers

Auto mode must enforce these limits per session. If any limit is hit, stop and ask.

| Breaker | Threshold | What happens |
|---------|-----------|--------------|
| API calls per session | 500 | Stop, show count by tool, ask to continue |
| Credits spent per session | 80% of workspace budget | Stop, show spend summary |
| Content pieces modified per batch | 50 | Stop, confirm before processing rest |
| Consecutive errors | 3 | Stop, diagnose before retrying |
| File overwrites in single session | 10 | Stop, show list of files changed |
| Cross-workspace writes | 1 (any) | Hard stop вҖ” never auto-approve writing outside active workspace |
| Publishing to >3 channels simultaneously | 1 | Stop, confirm distribution plan |

If a circuit breaker fires, log it in `logs/auto-audit.md` with full context and switch to interactive mode for the remainder of that workflow.

### Rollback safety

Before any multi-step auto-mode chain (brief вҶ’ write вҶ’ review вҶ’ publish), create a git checkpoint:
- `git add -A && git commit -m "AUTO checkpoint: before [workflow name]"`
- If the chain fails or a circuit breaker fires, the user can `git revert` to the checkpoint
- Log the checkpoint commit hash in `logs/auto-audit.md`

**Toggling:**
- Set during onboarding, or change anytime in `workspace.config.md`
- `**Execution mode:** auto` or `**Execution mode:** interactive`
- Can also toggle mid-session: just say "switch to auto" or "switch to interactive"

## Writing rules

1. **Every piece starts with a brief** вҖ” no writing without knowing the goal, audience, channel, and success metric
2. **Brand voice is law** вҖ” BRAND.md and TOV.md override personal style preferences
3. **Research before writing** вҖ” claims need sources, statistics need citations
4. **SEO is structural, not stuffed** вҖ” keywords guide structure, never forced into prose
5. **One piece, many channels** вҖ” everything written should be repurposable
6. **Measure what matters** вҖ” track performance against the brief's success metric, not vanity metrics
7. **Learn and iterate** вҖ” every debrief updates LEARNINGS.md

## Quality gates

Before any content is marked "approved":

1. **Brief check** вҖ” does it match the original brief?
2. **Brand check** вҖ” does it match BRAND.md and TOV.md?
3. **Audience check** вҖ” is it written for the right audience segment?
4. **SEO check** вҖ” target keyword, meta title, meta description, headings, internal links
5. **Readability check** вҖ” Flesch-Kincaid, sentence length, jargon level
6. **Source check** вҖ” all claims cited, no fabricated statistics
7. **Channel check** вҖ” formatted correctly for the target channel

## Learnings system

LEARNINGS.md is persistent intelligence. Load it before writing or strategizing. Update it after every debrief.

Categories:
- **Audience learnings** вҖ” what resonates, what doesn't
- **Channel learnings** вҖ” best times, formats, engagement patterns
- **SEO learnings** вҖ” what ranks, what doesn't, algorithm changes
- **Content type learnings** вҖ” which formats perform best
- **Topic learnings** вҖ” which themes drive engagement
- **Distribution learnings** вҖ” what amplification works
- **Anti-learnings** вҖ” things that definitely don't work
